\section{Implementation Details}

This section describes the practical realization of ADRDS, focusing on how the system components are instantiated and integrated while maintaining safety, modularity, and real-time responsiveness. The implementation prioritizes reproducibility, extensibility, and minimal operational overhead to support both experimentation and demonstration scenarios.

\subsection{Behavioral Simulation Engine}

The behavioral simulation engine is implemented as a modular, parameter-driven component that emulates ransomware-like activity without executing real malicious code. Ransomware behavior is represented using a state-driven execution model aligned with the behavioral taxonomy described in Section~IV. Each simulated stage emits time-stamped host-level events corresponding to file system operations, process activity, and resource utilization.

Simulation parameters such as execution intensity, file selection strategy, and encryption rate can be dynamically adjusted to model different ransomware variants and attacker strategies. This design allows ADRDS to safely reproduce realistic behavioral patterns while avoiding destructive actions such as actual file encryption.

\subsection{Feature Extraction and Preprocessing}

Raw behavioral events generated by the simulator are transformed into structured feature representations through a dedicated preprocessing pipeline. This pipeline aggregates low-level telemetry into time-windowed feature vectors that capture both instantaneous and temporal characteristics of system activity.

Extracted features include file access frequency, write-to-read ratios, entropy variation indicators, process-level resource usage, and burst-based activity metrics. Feature normalization and scaling are applied to ensure compatibility across machine learning models and to reduce sensitivity to system-specific variations.

\subsection{Machine Learning Models}

ADRDS employs a hybrid detection strategy that combines supervised classification with unsupervised anomaly detection. Supervised models, including Random Forests and gradient-boosted decision trees, are trained on labeled datasets derived from BigMalware and selected subsets of EMBER. These models are optimized for tabular behavioral features and provide reliable detection of known ransomware patterns.

To detect previously unseen or evolving ransomware variants, ADRDS integrates an unsupervised LSTM-based autoencoder trained exclusively on benign behavior sequences. During inference, reconstruction error is used as an anomaly score, enabling the identification of deviations from normal system behavior.

\subsection{Risk Scoring and Decision Logic}

Detection outputs from the supervised classifier and the anomaly detector are combined through a weighted risk fusion mechanism. This produces a unified risk score that reflects both known malicious patterns and behavioral novelty. Configurable thresholds determine whether activity is classified as benign, suspicious, or malicious, enabling adaptive sensitivity control.

This fusion-based decision logic supports early-stage detection and reduces false positives by leveraging complementary strengths of supervised and unsupervised learning.

\subsection{Visualization and User Interaction}

The visualization layer provides real-time feedback on system behavior and detection outcomes through an interactive dashboard. Behavioral timelines, feature distributions, risk scores, and classification confidence levels are rendered dynamically to support intuitive analysis.

User interactions with the dashboard allow on-the-fly adjustment of simulation parameters and detection thresholds. These interactions are propagated back into the simulation and detection pipeline, enabling iterative experimentation and immediate observation of system responses.

\subsection{System Integration and Runtime Characteristics}

All ADRDS components communicate through lightweight in-memory data structures to minimize latency and ensure near real-time operation. The modular architecture enables individual components to be independently modified or extended without affecting the overall system. Experimental evaluation indicates that the end-to-end pipeline operates with sub-second processing latency, making it suitable for interactive demonstrations and exploratory analysis.
